# Monolithic Architecture vs Microservices Architecture

## Monolithic Architecture — 모놀리식 구조

### 개념

하나의 애플리케이션이 **단일 프로세스·단일 배포 단위**로 동작하는 전통적인 구조. 사용자 인터페이스, 비즈니스 로직, 데이터 접근 레이어가 하나의 코드베이스에 통합되어 있으며 하나의 프로세스로 실행된다.

```
┌──────────────────────────────────────────┐
│              단일 애플리케이션            │
│                                          │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │  주문    │  │  결제    │  │  회원  │ │
│  │  모듈    │  │  모듈    │  │  모듈  │ │
│  └──────────┘  └──────────┘  └────────┘ │
│            공유 데이터베이스              │
└──────────────────────────────────────────┘
         ↓ 하나의 프로세스로 빌드 & 배포
```

### 모놀리식 구조의 장점

- **개발 초기 단순함**: 단일 코드베이스라 개발·디버깅·테스트가 직관적
- **낮은 운영 복잡도**: 배포 단위가 하나이므로 인프라 구성이 단순
- **모듈 간 통신 비용 없음**: 함수 호출 수준의 낮은 레이턴시
- **트랜잭션 처리 용이**: 단일 DB를 사용하므로 ACID 트랜잭션 적용 쉬움

### 모놀리식 구조의 한계

#### 1. 배포 위험 증가

코드의 어느 한 부분을 수정해도 **전체 애플리케이션을 다시 빌드하고 배포**해야 한다. 하나의 기능 변경이 전혀 관련 없는 모듈에 장애를 일으킬 수 있다.

```
결제 모듈 버그 수정
    → 주문·회원·알림 모듈 포함 전체 재배포 필요
    → 배포 중 서비스 전체 중단 (downtime)
```

#### 2. 장애 전파 (단일 장애 점)

한 모듈의 메모리 누수나 무한 루프가 전체 프로세스를 죽인다.

```
알림 모듈 OOM
    → 전체 프로세스 크래시
    → 주문·결제·회원 모두 중단
```

#### 3. 기술 스택 고착

전체 애플리케이션이 하나의 언어·프레임워크로 묶여 있어, 특정 모듈에 다른 기술을 적용하기 어렵다.

#### 4. 확장의 비효율

트래픽이 특정 모듈(예: 결제)에 몰려도 전체 애플리케이션을 통째로 스케일 아웃해야 한다.

```
결제 모듈에만 부하 집중
    → 전체 애플리케이션 인스턴스 3개로 증설
    → 주문·회원 모듈도 불필요하게 3배 확장 → 자원 낭비
```

#### 5. 코드베이스 비대화

서비스가 성장할수록 코드베이스가 커져 빌드 시간이 길어지고 팀 간 코드 충돌이 잦아진다.

## Microservices Architecture — 마이크로서비스 구조

### 개념

애플리케이션을 **독립적으로 배포 가능한 작은 서비스들의 집합**으로 구성하는 아키텍처. 각 서비스는 단일 비즈니스 기능을 담당하고, 자체 프로세스로 실행되며, 경량 API(주로 HTTP/REST 또는 gRPC)로 통신한다.

```
           API Gateway
               │
    ┌──────────┼──────────┐
    ↓          ↓          ↓
┌────────┐ ┌────────┐ ┌────────┐
│ 주문   │ │ 결제   │ │ 회원   │
│서비스  │ │서비스  │ │서비스  │
│  DB    │ │  DB    │ │  DB    │
└────────┘ └────────┘ └────────┘
   :3001      :3002      :3003
```

각 서비스는:
- 독립된 코드베이스와 저장소를 가짐
- 독립적으로 빌드·테스트·배포 가능
- 다른 서비스와 느슨하게 결합(Loosely Coupled)

### 마이크로서비스 구조의 장점

#### 1. 독립 배포

주문 서비스를 배포할 때 결제·회원 서비스는 영향받지 않는다. 팀별로 자신의 서비스를 독립적으로 릴리즈할 수 있어 **배포 빈도가 높아지고 위험이 줄어든다**.

#### 2. 장애 격리

결제 서비스가 다운돼도 주문 조회나 회원 기능은 계속 동작한다. (Circuit Breaker 패턴으로 더 강화 가능)

#### 3. 서비스별 독립 확장

트래픽이 집중된 서비스만 선택적으로 스케일 아웃.

```
결제 서비스만 Pod 1개 → 5개로 확장
주문·회원 서비스는 그대로 유지 → 자원 효율
```

#### 4. 기술 다양성 (Polyglot)

각 서비스가 가장 적합한 기술 스택을 선택할 수 있다.

```
주문 서비스  → Java / PostgreSQL
결제 서비스  → Go / MySQL
추천 서비스  → Python / Redis
```

#### 5. 팀 자율성

Conway's Law: "조직의 커뮤니케이션 구조가 시스템 설계에 반영된다." 마이크로서비스는 각 팀이 자신의 서비스를 완전히 소유(Own)할 수 있게 한다.

## 장단점 비교

| 항목 | 모놀리식 | 마이크로서비스 |
|------|----------|----------------|
| **배포 단위** | 전체 애플리케이션 | 개별 서비스 |
| **배포 위험** | 높음 (전체 영향) | 낮음 (서비스 범위 한정) |
| **장애 범위** | 전체 서비스 | 해당 서비스만 |
| **확장 단위** | 전체 앱 | 서비스 단위 |
| **기술 스택** | 단일 | 서비스별 선택 가능 |
| **초기 개발 속도** | 빠름 | 느림 (인프라 복잡도) |
| **운영 복잡도** | 낮음 | 높음 (서비스 수만큼) |
| **서비스 간 통신** | 함수 호출 | 네트워크 호출 (지연 발생) |
| **트랜잭션 처리** | 단순 (단일 DB) | 복잡 (Saga 패턴 등 필요) |
| **적합한 단계** | 초기 스타트업, 소규모 | 성장 단계, 대규모 팀 |

### 언제 마이크로서비스로 전환하는가

```
모놀리식으로 시작 → 아래 신호가 나타나면 분리 검토:

- 배포 주기가 길어지고 롤백이 잦아짐
- 특정 기능만 집중적으로 확장이 필요함
- 팀이 커져 코드 충돌과 조율 비용이 증가
- 특정 모듈에 다른 기술 스택이 필요해짐
```

> "모놀리식 지옥(Monolithic Hell)"에 빠지기 전에 분리하되, 너무 일찍 분리하면 "분산 시스템 복잡도 지옥"에 빠진다. 마틴 파울러는 **"처음에는 모놀리식으로 시작하라(Monolith First)"** 를 권장한다.

## 마이크로서비스에서 컨테이너가 주목받는 이유

### 문제: 마이크로서비스의 배포 복잡도

서비스가 10개, 100개가 되면 각 서비스마다 런타임 환경(언어 버전, 의존성 라이브러리)이 다르다. 서버에 직접 설치하면 충돌이 발생하고 환경 재현이 어렵다.

```
서버 1대에 여러 서비스 설치 시:
  주문 서비스  → Python 3.11, requests 2.28
  추천 서비스  → Python 3.9,  requests 2.20   ← 버전 충돌!
```

### 컨테이너가 해결하는 것

**컨테이너**는 애플리케이션 코드와 모든 의존성(런타임, 라이브러리, 설정)을 **하나의 격리된 패키지(이미지)** 로 묶어 어디서든 동일하게 실행할 수 있게 한다.

```
┌─────────────────────────────────────────┐
│              호스트 OS (Linux)           │
│                                         │
│  ┌──────────┐  ┌──────────┐  ┌───────┐ │
│  │ 주문     │  │ 결제     │  │ 회원  │ │
│  │ Python   │  │ Go 1.21  │  │ Node  │ │
│  │ 3.11     │  │          │  │ 20    │ │
│  └──────────┘  └──────────┘  └───────┘ │
│            Container Runtime            │
└─────────────────────────────────────────┘
```

| 컨테이너가 해결하는 문제 | 내용 |
|--------------------------|------|
| **환경 불일치** | "내 PC에서는 됐는데" 문제 해소. 이미지로 환경을 코드처럼 관리 |
| **의존성 충돌** | 각 컨테이너는 격리된 파일시스템 → 다른 서비스와 라이브러리 충돌 없음 |
| **빠른 시작/종료** | VM 대비 수 초 이내 기동 → 빠른 스케일 아웃/인 |
| **불변 배포 (Immutable)** | 이미지를 한 번 빌드하면 개발·스테이징·운영 환경에서 동일 동작 보장 |
| **자원 효율** | VM보다 오버헤드가 적어 하나의 호스트에 더 많은 서비스 실행 가능 |
| **선언적 배포** | Dockerfile + Kubernetes YAML로 배포 과정을 코드로 관리 (GitOps) |

### 마이크로서비스 + 컨테이너 + 쿠버네티스

컨테이너가 개별 서비스의 패키징 문제를 해결하면, **쿠버네티스(Kubernetes)** 는 수십~수백 개의 컨테이너를 클러스터에서 자동으로 배포·스케일·복구·로드밸런싱한다.

```
개발자                  컨테이너 레이어         오케스트레이션 레이어
코드 작성
    → Docker Build   →  이미지 (불변)
    → Docker Push    →  Registry 저장
    → kubectl apply  →  Kubernetes가
                        - 어느 노드에 배치할지 결정 (Scheduling)
                        - 장애 Pod 자동 재시작 (Self-healing)
                        - 트래픽에 따라 Pod 수 자동 조절 (HPA)
                        - 무중단 롤링 배포 (Rolling Update)
```

> 결론: 마이크로서비스는 "무엇을 분리할까"를 정의하고, 컨테이너는 "어떻게 격리·패키징할까"를 해결하며, 쿠버네티스는 "분리된 서비스들을 어떻게 운영할까"를 해결한다. 세 기술이 함께 작동할 때 마이크로서비스 아키텍처의 장점이 실현된다.
